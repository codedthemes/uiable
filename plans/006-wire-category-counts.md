## Why this matters

The components landing page shows a per-category "N variants" count, read from a committed JSON file
that is generated by a script **no npm script ever runs**. So the counts can silently drift from the
actual registry: add or remove a variant, and the UI count stays wrong until someone remembers to run
the generator by hand. This plan adds an npm script for the generator and chains it into the existing
`registry:build` so the count file is always regenerated alongside the registry, then refreshes the
committed file.

- **No script runs the generator** — `package.json:7-8`:

  "registry:catalog": "node src/scripts/build-catalog.mjs",

- `project-docs/STRUCTURE_INCONSISTENCIES.md` item #4 documents this drift (it cites the old path

## Steps

### Step 1: Add the npm script and chain it

Edit `package.json` scripts:
1. Add: `"registry:counts": "node src/scripts/generate-counts.mjs",`
2. Change `registry:build` so counts regenerate as part of the registry build:
   `"registry:build": "shadcn build && npm run registry:catalog && npm run registry:counts",`

**Verify**: `npm run` lists `registry:counts`; `git diff package.json` shows exactly those edits.

### Step 2: Regenerate the committed counts file

Run `node src/scripts/generate-counts.mjs` (or `npm run registry:counts`).

**Verify**: exit 0; `git diff src/category-counts.json` shows the file rewritten to match the current
registry (it may be unchanged if it was already current — that's fine; if it changes, that change *is*
the drift this plan fixes).

### Step 3: Confirm the app still builds with the (possibly updated) counts

Run `npm run build`, then `npm run lint`.

**Verify**: both exit 0. The components page imports the JSON at build time, so a malformed file would
fail the build.

## Maintenance notes

- This wires counts into `registry:build` (the local/release registry workflow), not into CI. CI today
  runs only `npm i && npm run build` and does not regenerate the registry, so the committed
  `src/category-counts.json` remains the source the build consumes. If a future CI step regenerates the
  registry, add a check that `src/category-counts.json` is not stale (e.g. run the generator and fail
  if `git diff` is non-empty).
- The deeper fix (a single generated category source feeding counts, nav, and route params) is tracked
  in `plans/README.md` direction findings.
- Reviewer should confirm `registry:build` still runs `shadcn build` and `registry:catalog` first, with
  `registry:counts` appended last.
