## Findings considered and rejected (do not re-audit)

- **Path traversal via the dynamic-import slug** (`preview/[...slug]/page.tsx`, `category-view.tsx`):
  Turbopack/webpack compile a template-literal `import()` to a directory-scoped context module
  whose request regex does not match `../`. Arbitrary-file read is not reachable. Defense-in-depth only.
- **`dangerouslySetInnerHTML` fed by shiki `codeToHtml`**: trusted highlighter, repo-authored source
  as input. By-design, not a vulnerability.
- **"setState after unmount" cleanup nits** (`use-copy-to-clipboard.ts`, `customizer/index.tsx`):
  React 18 removed that warning; near-zero impact. (The genuine unhandled-promise-rejection nits are
  tracked as finding #11 "minor cleanups" but were not selected for a plan.)
- **`document as any` in the customizer**: has a runtime `startViewTransition` guard. Smell, not a bug.
- **Component-level memoization / LazySection placeholder sizing**: speculative without profiling; the
  per-item dynamic import is inherent to the catalog design.

## Newly discovered during execution (2026-06-17)

- **Unused `SearchBar` import in `src/app/(dashboard-layout)/layout.tsx`** — surfaced by ESLint during plan 005 execution. Pre-existing on `dev`; component is imported (line 17) and commented out (line 28). Not yet planned.
- **ESLint picks up `.claude/worktrees/` directories** — no `ignore` rule covers them; any nested worktree Next.js project gets linted and produces UNKNOWN errors. Fix: add `.claude/worktrees/` to `.eslintignore` or `eslint.config.js` ignores. S effort.
- **Stale `package-lock.json` surfaces ~11 `shadcn`-tree advisories on any install.** Found while
  executing plan 001: the committed lockfile audits clean-ish (2 advisories) but is stale relative to
  the `^`-ranged deps in `package.json` (notably `shadcn: ^4.11.0`). Any `npm install` reconciles the
  tree and pulls newer, advisory-carrying versions of `hono`, `@hono/node-server`, `express-rate-limit`,
  `path-to-regexp`, `qs`, `@babel/core`, `js-yaml`, etc. These are dev/registry-tooling deps, likely
  not in the deployed runtime, but the divergence between the committed lockfile and what installs
  resolve is a real maintenance hazard. Not yet planned — re-run `improve` to triage (likely an
  `npm dedupe` / lockfile-refresh + targeted overrides, scoped away from the deployed bundle).
